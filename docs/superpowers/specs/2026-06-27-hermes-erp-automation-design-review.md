## 🔴 치명적 (이건 통과 못 시킴)

### 1. 운영 부담을 정면으로 과소평가했다 — 가장 큰 리스크
지금 스택은 Vercel + Supabase, **제로옵스(zero-ops) 매니지드**다. 이걸 **raw VPS + Docker + Python 에이전트 프레임워크 상시 구동**으로 옮긴다는 건, 6명짜리(사실상 개발자 1명) 건설회사가 갑자기 떠안는 게:

- OS 패치, Docker 운영, 프로세스 감시/자동재기동, 로그 로테이션, 시크릿 관리, 업타임 모니터링, 백업/복구

"월 1만원대"는 **VPS 임대료**일 뿐이고, 진짜 비용은 **사람의 운영 시간과 장애 대응**이다. 문서 D5의 "자동화의 생명은 항상 켜짐"이 바로 이 리스크의 본질인데, **누가 이걸 24/7 켜진 상태로 유지하느냐**에 대한 답이 없다. (§13-5 "운영 담당" 미정 = 설계의 핵심 구멍)

### 2. 단일 장애점(SPOF)을 만들면서 "두뇌"를 거기에 올렸다
§14에 "VPS 단일 장애점"을 리스크로 적어놓고 해결책은 "백업·재기동 절차 필요"로 손 흔들기. 그런데 이 박스가 죽으면:

- 야간에 cron 보고 안 나감, 입금 처리 멈춤, 문지기 알림 안 울림 → **아무도 죽은 걸 모름**

회사의 "두뇌"를 HA·페일오버 없는 단일 박스에 올리는 건, 매니지드 서버리스(Vercel은 알아서 떠 있음)에서 **안정성을 일부러 떨어뜨리는 역행**이다.

### 3. 프로젝트가 이미 "비현실적"이라 결론 낸 걸 되살렸다 (공문 폼필드)
CLAUDE.md 철학 #9: **"서류는 Cowork → 구글드라이브 — 폼필드 유지보수 비현실적"**.
회사가 이미 폼필드 자동화는 못 한다고 결론 내고 Cowork+Drive로 우회한 사안인데, 이 설계 §7은 **"Hermes가 폼필드 의미기반 자동분석·매핑"**으로 그걸 정면으로 되살린다. 게다가 §13-2에서 HWP(한글) 포맷은 미해결 — HWP 프로그래밍 생성/필링은 악명 높게 어렵다. **기존에 학습된 교훈을 무시한 설계**라는 게 제일 위험한 신호다. §7 전체가 가장 야심차고 가장 검증 안 된 부분인데 핵심 자동화 가치로 잡혀 있다.

### 4. "완전 대체(방향 A)"의 정당성이 약하다 — 자기 강점을 버린다
문서 스스로 §1에서 현 구조를 **"공용 코어 + 멀티채널"**이라 자랑한다. 그런데 D2는 그 멀티채널 중 하나(인앱 사이드패널)를 **제거**한다. 모순이다.

- 인앱 비서는 **사용자가 보고 있는 ERP 화면 맥락 안에** 있다. Slack으로 옮기면 프로젝트 보다가 질문하려면 **컨텍스트 스위칭**해야 함.
- MCP 코어를 공유하면 Slack과 인앱을 **둘 다 유지**할 수 있는데, 굳이 제거할 이유는 "일원화"라는 미감뿐. **제거는 위험을 추가하지 가치를 더하지 않는다.** "병행 → 제거"가 아니라 "병행 유지"가 정답일 수 있다.

## 🟠 심각 (조건부 통과)

### 5. Obsidian git 양방향 동기화 = 머지 컨플릭트 시한폭탄
서버가 ingest로 쓰고, 데스크톱에서 사람이 큐레이션. **양쪽이 동시에 같은 마크다운을 수정** → git 충돌. 건설회사 직원이 git merge conflict를 풀까? `obsidian-headless`/git-sync는 단방향 읽기엔 괜찮아도 **양방향 편집 저장소로는 취약**하다. 동기화 모델을 다시 설계하거나, 서버 쓰기와 사람 큐레이션의 **소유권 경계(파일/디렉토리 단위 락)**를 명시해야 함.

### 6. 2계층 기억의 경계는 이론상 깔끔, 실제론 샌다
§8은 "거래 수치는 절대 위키에 안 넣음"이라지만, 정작 위키에 담는다는 게 **"거래처 평가·계절 패턴·현장 복기·단가 패턴"** — 이것들은 **본질적으로 거래 데이터에서 파생**된다. "A업체는 단가 후려치고 공기 지연"이라는 평가는 금액·일정 없이 못 쓴다. 경계는 반드시 흐려지고, §14가 인정한 "잘못된 학습이 사실로 굳음"은 자가학습 루프의 구조적 약점이다.

### 7. 개인정보(PII) × 자가학습 × git 히스토리 = 규정 리스크
접수 업무의 핵심 데이터가 고객 이름·연락처·주소(PII)다. 자가학습은 **실제 케이스**에서 배우는 거라 PII가 안 들어갈 수가 없음. 그런데 §9는 "민감정보는 DB에만 둘지 **검토**" — 미해결. **git은 삭제해도 히스토리에 영원히 남는다.** VPS의 git repo에 PII 축적 → 개인정보보호법 리스크. 이건 "검토 필요"가 아니라 **설계 전제로 못 박아야** 할 사안이다.

### 8. 인증이 god-mode 단일 토큰이다
§5.3 "Bearer 토큰 자체검증" + `service_role_key`. 즉 **Hermes가 가진 정적 토큰 하나 = DB 전체 쓰기 권한**. VPS 털리면 그 토큰으로 진실원천 DB를 통째로 조작 가능. 토큰 로테이션·MCP 레이어 사용자별 권한 스코핑·에이전트 발 쓰기의 감사로그 — 다 빠져 있다. Slack유저↔staff role 매핑은 §10에서 "초안, 확인 필요"로 희망사항 수준.

### 9. 안전장치가 "사람 확인카드"에 100% 의존 → 승인 피로
쓰기 도구는 Slack 확인카드 승인 후 실행(§9). 그런데 바쁜 6명이 카드 쏟아지면 **고무도장(rubber-stamp)** 찍는다. 특히 입력 소스가 **카톡 스크린샷의 Vision 파싱**(§6 Flow1)이라 garbage-in 가능성이 높은데, 사람은 대충 승인 → **잘못 파싱된 주소·입금이 진실원천을 오염**. 입금 중복체크(payments.ts)는 있어 다행이나, idempotency·dry-run·롤백 전략이 명시 안 됨.

### 10. 진짜 병목(접수 입력)은 자동화가 안 된다
D8/§12가 인정하듯 카카오톡 공개 API가 없어서, 자동화의 정문은 **사람이 텍스트/스크린샷을 Slack에 붙여넣기**다. 즉 가장 비구조적·오류 많은 단계(고객 → 입력)는 **여전히 수동 + AI 파싱 오류 레이어만 추가**. 자동화 ROI가 가장 큰 지점에서 이득이 한계적이다.

## 🟡 짚고 넘어갈 것

- **11. 비용 모델 왜곡**: "월 1만원대"는 VPS만. 상시 에이전트 루프 + 보고 4종 + 문지기 알림 + 개인별 모델링 + Vision 파싱 = **Claude API 토큰이 진짜 월비용**. 토큰 예산 분석이 0.
- **12. 프레임워크 베팅**: 회사 핵심 자동화를 신생 MIT 프로젝트(Nous hermes-agent)에 게이트웨이·cron·스킬·기억까지 **밀착 결합**. 중단/브레이킹 체인지/보안홀 시 탈출구 없음. SLA 없는 의존성.
- **13. 마이그레이션 출구기준 부재**: P6 "검증 완료 후 제거"의 **검증 완료 정의가 없음**. 성공 지표·롤백 플랜 없이 옛 시스템을 떼면 되돌릴 수 없음.
- **14. 관측가능성 없음**: 비결정적 LLM 에이전트가 원격 VPS에서 잘못된 입금매칭·단계전환을 했을 때 **디버그·리플레이 스토리가 없음**. 금전·공문 결정을 하는 시스템인데.
- **15. 버스팩터 = 1**: 설계·운영이 사실상 1인. VPS 재기동, git 충돌 해결, 토큰 로테이션, 위키 스키마 갱신 — 그 1명 부재 시 멈춤. **시스템 정교함이 조직의 유지보수 능력을 초과**한다. (이게 메타 리스크)
- **16. 현장직 Slack 채택**: 한국 현장 직원이 KakaoTalk 두고 Slack DM을 업무창구로 쓸까. 채택 저항 리스크.

## 한 줄 총평
**"항상 켜진 자동화"라는 목표는 맞지만, 그 대가로 매니지드 제로옵스를 버리고 6명 조직이 감당 못 할 운영·보안·SPOF 부채를 떠안는 설계다.** 그리고 회사가 이미 "비현실적"이라 결론 낸 폼필드 자동화를 되살린 점, 멀쩡한 인앱 채널을 굳이 제거하는 점은 자기모순이다.

---

## 통과시키려면 (우선순위)

1. **상시구동을 VPS 자체호스팅이 아니라 매니지드로** — Hermes를 컨테이너 PaaS(Fly.io/Railway/Render 등)나, cron은 이미 있는 Vercel/Supabase 스케줄러 활용 검토. 운영 부담·SPOF·백업이 한 번에 해결.
2. **"완전 대체" → "채널 추가"로 완화.** 인앱 비서 유지하고 Slack을 MCP 코어 위에 **추가** 채널로. 제거는 나중에 데이터로 정당화될 때만.
3. **§7 공문 폼필드 자동화는 설계에서 분리.** 검증 안 된 R&D 트랙으로 빼고, 메인 자동화에서 의존성 끊기. 기존 Cowork+Drive 결론 존중.
4. **PII·git 히스토리 문제를 전제로 확정** — 위키는 PII 제로, 익명 참조키만. "검토"로 남기지 말 것.
5. **MCP 인증을 사용자 스코프 토큰 + 감사로그로.** service_role god-token 단일화 금지.
6. **P6 제거 기준(성공 지표 + 롤백)**을 숫자로 정의.

## 그래도 잘된 점 (균형)

- "대체 = 로직 폐기 아님"(D3): 도구 핸들러를 MCP로 재사용한다는 판단은 옳다. payments.ts 등 검증된 코어 보존.
- 진실원천(Supabase)과 해석지식(위키)을 **개념적으로** 분리하려는 의도는 건전.
- 시청 공문 **자동 제출 금지**(고위험 차단)는 올바른 안전 경계.
- 무중단 단계적 이행(P0~P6) + 병행 운영(P2) 프레임은 방향이 맞다 — 출구기준만 채우면 됨.
- `validateProjectData()` 저장 전 검증 유지(D, §5.3)는 DB 오염 방어로 적절.
