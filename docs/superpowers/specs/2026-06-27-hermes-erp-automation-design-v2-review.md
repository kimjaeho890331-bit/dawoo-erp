## 0. 공문 폼필드 (D10 / §5.6) — "feasible 확정" 주장 검증

v1 리뷰 #3("이미 비현실적이라 결론 낸 폼필드를 되살렸다")의 근거는 CLAUDE.md 철학 #9 한 줄이었는데, 실제 docs를 확인하니 **그 단정은 살짝 과했다.** 정확한 사실관계:

- `AGENT.md` §52-56: 회사는 **"양식 올리면 AI가 폼필드 자동 분석 → 확인 → 확정"**을 Phase 2 서류팀 도구로 **이미 계획**해뒀다.
- `DOCUMENTS.md` §36-42: "재호가 빈 양식 올림 → **Claude가 폼필드 추가** → 확인 → Storage 저장 → 데이터 자동 입력"이 정식 흐름.
- CLAUDE.md #9 "폼필드 유지보수 비현실적"이 말한 건 **사람이 15시×문서×연도 폼필드를 손으로 유지**하는 게 비현실적이란 뜻 → 그래서 Cowork+Drive로 우회.

즉 회사 입장은 **"폼필드 자동화 자체가 불가능"이 아니라 "사람 손 유지보수가 불가능"**이다. v2의 HWPX 런타임 채우기는 **사람 유지보수를 빼고 엔진이 채우는** 제3의 길이라, 기존 결론과 충돌하지 않고 오히려 결을 같이한다. **→ v1 지적 #3은 v2에서 유효하게 해소됨.**

**다만 "feasible(PoC 확인)"이라는 표현은 아직 과하다.** 실제 PoC 증거(어떤 양식·정확도)가 문서에 없다. 기술 경로 자체는 타당하다 — HWPX는 zip+XML(OWPML)이라 **채우기(fill)는 리눅스에서 한글 없이 가능**, 변환(hwp→hwpx)만 한글/LibreOffice 필요하다는 분리도 정확하다. 진짜 위험은 거기가 아니라:

- **"의미기반 매핑"의 강건성**: 시별 템플릿은 표 구조·병합셀·중첩표가 제각각. XML 텍스트 노드 매핑은 1종에선 되지만 15시 매트릭스에서 깨지기 쉬움.
- **P5 성공지표가 "수원 양식 1종 정확 채움"(§12)** — PoC지 §7의 `[15시]×[문서]×[연도]` 변동성을 푼 게 아님. **1종 성공 ≠ 매트릭스 해결.** 지표가 난이도를 과소표현.

→ **권고**: D10의 "feasible 확정"을 **"기술 경로 타당, P5 진입 전 실제 수원 .hwpx 1종 PoC(정확도·표구조 깨짐 측정) 필수"**로 톤다운. 디커플(§5.6)은 잘 했으니 유지.

## 🔴 v2에서 새로 생긴 / 안 닫힌 구멍

### N1. §7 ↔ §8 숫자 정책 정면충돌 (실제 모순)
- §8: 위키엔 **"PII·숫자 금지"**, 숫자는 DB만.
- §7 line 127: 연도별 규칙 **"지원률·자격·단가"를 위키 `concepts/지원사업/{연도}/{시}.md`에** 둠.

**지원률·단가는 숫자다.** 한 문서 안에서 "숫자 위키 금지"와 "단가를 위키에"가 동시에 존재.
→ **해소**: §8을 **"거래 데이터(특정 프로젝트의 금액·잔액·일정)는 금지 / 공개 정책수치(지원률·표준단가 등 사실상 공지된 규칙)는 허용"**으로 구분. 지금 표현으론 큐레이터가 뭘 어디 둘지 매번 헷갈림.

### N2. PII를 Obsidian에서 막았더니 텔레그램으로 옮겨갔다
v1 지적(PII×git)은 §5.5에서 잘 닫았는데, D6/D9로 **텔레그램 '고객접수' 단톡방**이 신설되면서 PII 표면이 거기로 이동:

- 봇 **privacy mode off** = 방의 **모든 메시지(고객 이름·연락처·주소)를 봇이 수신** → 텔레그램(해외 서버)에 히스토리 잔류.
- §10 "PII는 즉시 Supabase로, 방은 전송 통로"라지만 **텔레그램 서버는 원문을 보관**. git 영구잔류를 걱정했던 논리가 그대로 텔레그램에 적용.
- 접근통제도 "방은 비공개·최소인원"이라는 **사회적 약속**이지 기술적 통제가 아님. 한 명 잘못 초대 시 PII 노출.

→ **권고**: (a) 접수방은 **구조화 입력(봇 슬래시커맨드/폼) 유도**로 자유텍스트 PII 최소화, (b) 처리 후 **봇이 원문 메시지 삭제(deleteMessage)**, (c) PII 보존정책을 텔레그램에도 명시 적용. "Obsidian PII 제로"만큼 텔레그램도 닫아야 일관됨.

### N3. 매니지드 = scale-to-zero면 cron이 죽는다 (구체적 함정)
D5의 매니지드 전환(Fly/Railway/Render)은 운영부담·SPOF 측면에서 정답. 단 **이 플랫폼 저가 티어는 유휴 시 scale-to-zero(잠듦)가 기본**인 경우가 많음. hermes-agent의 **cron·게이트웨이는 프로세스가 항상 떠 있어야** 작동 → 잠들면 **정시 보고/알림 미발사·텔레그램 수신 지연**.
→ **권고**: "월 1만원대"가 아니라 **always-on 인스턴스(scale-to-zero 비활성) 티어**를 §11/§14-3에 명시.

### N4. "스코프 토큰"이 아직 명목상 (enforcement 미정)
§5.3/§10에서 god 토큰을 버린 방향은 옳음. 그러나 **스코프 기준**(도구별 capability? 사용자별?)이 미정. 에이전트는 어차피 register/deposit 쓰기 권한 필요 → 컨테이너 침해 시 **스코프 토큰으로도 그 범위 쓰기는 가능**. 감사로그는 탐지지 예방이 아님.
→ **권고**: ① `agent_actions` 기록을 MCP 코드가 아니라 **DB 트리거로 강제**(에이전트 우회 차단), ② 쓰기/읽기 토큰 분리, ③ 고위험 쓰기(입금·단계전환)는 토큰만으로 불가, **관리자 승인 게이트를 도구 레벨에서 강제**.

### N5 (잔여). 2계층 경계 누수는 완화됐으나 소멸은 아님
"참조키만, 숫자는 DB"(§8)로 크게 개선. 다만 "A거래처 단가 후려치고 공기 지연"식 **판단은 허용**인데 이는 숫자에서 파생된 준-거래정보. confidence/contested 플래그(§15) 방향은 맞음 — **N1만 정리하면 운용 가능 수준.**

## 2. 한 줄 총평
**v2는 v1의 거의 모든 치명타를 제대로 닫은, 통과 가능 궤도의 설계다.** 채널 2계층(직원 텔레그램 / 관리자 Slack)과 관측 대시보드(§9)는 v1 대비 확실히 성숙. 남은 건 큰 재설계가 아니라 **4개의 마감**.

## 3. 통과 조건 (이것만 마감하면 writing-plans 진입)
1. **§7↔§8 숫자 정책 모순 해소** — 정책수치 허용 / 거래수치 금지로 재정의 (N1)
2. **텔레그램 PII 표면 닫기** — 구조화 입력 + 원문 삭제 + 보존정책 명시 (N2)
3. **always-on 티어 명시** — scale-to-zero면 cron 사망 (N3)
4. **D10 "feasible 확정" → "PoC 후 확정"으로 정직화** + 스코프 토큰 enforcement 구체화 (§0, N4)

## 4. 잘된 점 (균형)
- 채널 2계층(D6): 직원은 학습 0(텔레그램 유지), 관리자만 Slack. v1 "현장직 Slack 채택" 리스크를 영리하게 해소.
- 관측 대시보드 §9(감사테이블 + 원본↔결과 대조 + 승인큐): 비결정적 에이전트 운용의 정석. v1 최대 공백 메움.
- 이행 계획 §12에 단계별 성공지표·롤백 명시: 출구기준 부재 해소.
- 공문 트랙 디커플(§5.6): 검증 안 된 R&D가 메인 자동화를 막지 않게 분리. 올바른 의존성 관리.
- 인앱 제거 보류(D2/P6): "데이터로 정당화될 때만 제거"는 멀쩡한 채널을 미감으로 버리던 v1 모순 해소.
